# Codex + MCP·스킬 조합으로 매칭신청서 작성 어시스턴트를 3일 만에 개발·배포하기

> **작성자:** 강용진
> **날짜:** 2026-02-28
> **에피소드:** EP2 (2주차)

---

## Situation (상황)

- **문제 상황:** "매칭신청서 작성 어시스턴트"(AI 챗봇 기반 풀스택 기능)를 3일 안에 개발·배포해야 했다. UI/UX 전문가가 아니고 프론트엔드에 익숙하지 않은 상태에서, 채팅 UI부터 AI 챗봇 로직·DB 스키마까지 풀스택을 빠르게 완성해야 했다. 에이전틱 AI를 별 다른 세팅 없이 사용한다면 AI와 핑퐁을 여러 번 해야 하고, 결과물이 마음에 안 들어 수정을 반복하면 3일 안에 원활히 끝낼 수 없다고 판단했다.
- **사용 도구:** Codex (codex-5.3-max) + Context7, Sequential Thinking, 웹 브라우징 MCP, Desktop Commander, Frontend Design Skill
- **배경:** 이번엔 "처음부터 고성능 모델 + MCP + 스킬을 풀세팅해서 한 번에 제대로 뽑자"는 접근으로 시작했다.

## Task (과제)

- **목표:** 3일 만에 아래 조건을 충족하는 기능을 개발해 배포한다.
    1. 기능 버그가 없을 것
    2. AI 챗봇의 대화가 목표 달성(매칭신청서 작성 완료) 방향으로 매끄럽게 이어질 것
    3. UI가 잘 갖춰져 있을 것

## Action (행동)

### 시행착오

프론트엔드·백엔드·DB 설계 단에서는 큰 시행착오가 없었다. 처음부터 Codex + MCP + Frontend Design Skill 조합으로 접근했기 때문이다. 

**시행착오가 집중된 곳은 AI 챗봇의 프롬프트 엔지니어링**이었다.

- AI 챗봇이 이미 물어본 질문을 다시 묻거나, 질문 순서를 지키지 않았다.
- 답변 퀄리티가 낮았다 (예: 월 예산 추천 시 범위를 지나치게 넓게 제시).
- 전반적으로 대화가 매끄럽지 못해, 유저 관점에서 목표 달성까지의 흐름이 끊겼다.

### 해결 방법

**1. Codex Plan 모드로 기획·설계부터 시작**

Codex를 Plan 모드로 전환해, MCP·스킬을 활용하여 아래 순서로 진행했다.

1. **기획 문서 작성:** 문제 → 근본 원인 → 솔루션 → 가설로 구성된 기획 문서를 먼저 작성.
2. **개발 계획서 작성:** Plan 모드에서 풀스택 개발 계획서를 산출.
3. **UI/UX 레퍼런스 제공:** 참고할 채팅 UI 레퍼런스를 직접 보여줘, 디자인 방향을 잡아줌.

**2. MCP·스킬 풀세팅**

- **Context7:** 라이브러리 문서 실시간 조회로 정확한 코드 생성.
- **Sequential Thinking:** 복잡한 기능을 단계별로 사고하게 함.
- **웹 브라우징 MCP:** 레퍼런스 확인·실시간 정보 탐색.
- **Desktop Commander:** 로컬 환경 제어.
- **Frontend Design Skill:** 채팅 UI 설계·구현 가이드.

**3. 프롬프트 개선**

AI 챗봇은 기획서와 계획서를 기반으로 기능의 목적은 이해했지만, 실제 고객을 대면하는 상담원 관점의 도메인 지식이 부족했다.

- 고객이 월 예산을 결정하려면 먼저 어떤 정보(업무 목표, 현재 상황, 업무 범위)를 받아야 하는지 몰랐다.
- 매칭신청서 항목을 채워나가는 순서를 모르니, 질문 흐름이 뒤죽박죽이었다.
- "이 직무에 주 투입시간 n시간이면 월 예산 얼마가 적합하다" 같은 현실적인 기준 데이터가 없어, 추천 범위가 지나치게 넓었다.

그래서 AI 챗봇에 대화를 어떤 순서로 이끌어가야 하는지 가이드를 프롬프트에 추가하고 참고 데이터(예: 직무별 주 투입시간별 평균 대금)를 제공하며 문제점 피드백 → 프롬프트 수정 → 재테스트 사이클을 반복하여 개선했다.

- 애초에 구체적인 설명이 필요했던 기능인데 내가 구체적인 설명을 안해줘서 시행착오가 생기지 않았나?라는 생각이 들었다. 

## Result (결과)

- **채팅 UI:** 레퍼런스와 Frontend Design Skill 덕분에 단번에 잘 만들어졌다. 이후 세부 조정 외에 큰 수정이 없었다.
- **AI 챗봇 로직·DB 스키마:** Plan 모드에서 설계한 대로 잘 구현되었다. 이후 수정은 프롬프트 개선뿐이었다.
- **AI 챗봇 대화 품질:** 대화 순서 가이드를 프롬프트에 추가하고 참고 데이터(직무별 주 투입시간별 평균 대금 등)를 제공하자, 질문 순서·추천 정확도·대화 흐름이 눈에 띄게 개선되었다.
- **3일 내 개발·배포 완료.** 기능 버그 없이 목표대로 배포했다.

## Learning (배움)

- **결정적 활용팁:** "기존대로 하면 핑퐁이 많아질 것 같다"고 느낀다면, 고성능 모델 + MCP + 스킬을 풀세팅하고 Plan 모드에서 기획·설계를 먼저 산출한 뒤 개발에 들어가는 것이 훨씬 효율적이다. 한 번에 제대로 뽑으려는 준비가 핑퐁 횟수를 줄인다.
- **배운 것:**
    - AI 기반 기능을 만들려면, 그 기능의 도메인을 우리가 먼저 잘 알아야 하고 잘 알려줘야 한다. AI는 기획서만으로 "기능의 목적"은 파악하지만, "고객에게 어떤 순서로 무엇을 물어야 하는지", "어떤 기준 데이터로 추천해야 하는지" 같은 현장 지식은 모른다. 대화 흐름 가이드와 참고 데이터(예: 직무별 주 투입시간별 평균 대금)를 사전에 제공해야 제대로 된 AI 기능이 나온다.
    - Codex는 MCP 연동이 간편하고, 한 번 연결하면 끊기지 않아 안정적이다. 컨텍스트 압축도 잘 되어 대화 세션이 오래 유지되며, 토큰 효율도 높다.
    - 단점으로는 개발 수행 속도가 Cursor 대비 느리고, 변경 사항을 살펴보고 롤백하는 UX가 Cursor보다 불편했다.
    - AI 챗봇 테스트·피드백 사이클을 AI로 자동화할 수 있다면 개발 속도와 품질을 더 끌어올릴 수 있지 않을까라고 생각이 들었다.
