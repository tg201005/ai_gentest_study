# plan-dev 스킬, 병렬 실행으로 개선하려다 워크플로우 구조를 통째로 바꾼 이야기

> **작성자:** 안선정
> **날짜:** 2026-03-14
> **에피소드:** EP4 (4주차)

---

## Situation (상황)

- **문제 상황:** plan-dev 스킬이 Notion 기획서 읽기 → Figma 파일 확인 → 프로젝트 내용 확인 → 구현 계획 세우기 순서로 순차 실행되어, 다른 스킬들에 비해 압도적으로 시간이 오래 걸렸다.
- **사용 도구:** Claude Code, plan-dev 스킬 (Notion + Figma MCP + 프로젝트 구조 분석 자동화)
- **배경:** plan-dev 스킬은 기획서를 읽고, Figma 디자인을 분석하고, 기존 프로젝트 코드를 탐색한 뒤 구현 계획 문서를 만들어주는 스킬이다. 작업 자체는 편했는데 실행할 때마다 체감 대기 시간이 상당했다. "어차피 독립적인 작업들인데 병렬로 돌리면 빨라지지 않을까?"라는 생각에서 개선을 시작하게 됐다.

## Task (과제)

- **목표:** plan-dev 스킬의 4단계 순차 워크플로우 중 독립적으로 수행 가능한 단계를 병렬 실행으로 전환하여 전체 소요 시간을 단축한다.

## Action (행동)

### 시행착오 1 — 일단 Figma + 프로젝트 탐색을 병렬로 묶어봤다

처음에는 단순하게 생각했다. Notion 분석이 끝나면 이후 단계인 Figma 디자인 분석과 프로젝트 구조 탐색은 서로 의존성이 없으니, 이 둘을 Agent tool로 동시에 실행시키면 되겠다 싶었다.

```
AS-IS: Notion 분석 → Figma 디자인 분석 → 프로젝트 구성 분석 → 계획 수립
TO-BE: Notion 분석 → (병렬) Figma 디자인 분석 / 프로젝트 구성 분석 → 계획 수립
```

스킬 프롬프트에 아래와 같은 지시를 추가했다.

```
3단계: 병렬 분석 (Figma + 프로젝트 구조 동시 실행)

Agent tool을 사용해 아래 두 작업을 동시에 실행하세요. 두 작업은 서로 독립적이므로 반드시 병렬로 처리합니다.

두 Agent 모두 2단계에서 파악한 기능명, 핵심 기능 목록, 데이터 모델을 컨텍스트로 전달받아
관련 정보를 중점적으로 분석합니다.

Agent A — Figma 디자인 분석
분석 포커스: 기획서의 핵심 기능과 화면 흐름을 기준으로 관련 노드를 중점 분석합니다.
기획서와 무관한 노드는 간략히 처리합니다.

Agent B — 프로젝트 구조 탐색
탐색 포커스: 기획서의 기능명·데이터 모델 키워드를 기반으로 관련 파일을 우선 탐색합니다.
```

결과는 "체감상 약간 빨라진 것 같기도 한" 수준이었다. 두 작업이 동시에 돌아가는 건 눈으로 확인됐지만, 병목이 Figma 분석 쪽에 몰려 있어서 전체 시간이 크게 줄지는 않았다.

**문제의 원인을 파고들어보니:** Figma 파일을 여는 시점에 Agent는 아직 "어떤 화면이 실제로 바뀌었는지" 모르는 상태였다. 그러다 보니 기획서와 관계없는 화면 노드까지 넓게 탐색하고, 정작 수정이 필요한 노드는 얕게만 가져오는 상황이 반복됐다.

### 시행착오 2 — 생각해보니 Notion 분석이 반드시 먼저일 이유가 없었다

여기서 한 발짝 물러서서 워크플로우 전체를 다시 봤다.

- Notion 분석: 기획 내용을 파악해 "무엇을 만들지" 결정
- 프로젝트 구조 탐색: "현재 코드가 어떻게 되어있는지" 파악
- Figma 분석: "화면이 어떻게 바뀌어야 하는지" 파악

Notion 분석과 프로젝트 구조 탐색은 서로 의존성이 없다. 오히려 이 두 가지를 먼저 병렬로 끝내고, 그 결과를 컨텍스트로 넘겨줘야 Figma 분석이 "관련된 화면만" 집중해서 볼 수 있다는 걸 깨달았다.

```
AS-IS: Notion 분석 → (병렬) Figma 디자인 분석 / 프로젝트 구성 분석
TO-BE: (병렬) Notion 분석 / 프로젝트 구성 분석 → Figma 디자인 분석
```

수정 포인트는 두 가지였다.

1. **Notion 분석과 프로젝트 탐색을 1단계에서 병렬로 실행하도록 변경**
   - 이 단계에서 "기능명, 핵심 기능 목록, 데이터 모델, 수정 대상 화면 키워드"를 확보
2. **Figma 분석 Agent에 1단계 결과를 컨텍스트로 전달**
   - "이 기능과 관련된 화면만 찾아라. 무관한 노드는 무시해라"는 지시를 명확히 추가

## Result (결과)

- Figma API 호출 횟수가 체감상 10회 수준에서 3회 수준으로 줄었다. 탐색을 "넓게 얕게"에서 "좁게 깊게"로 바꾼 덕분이다.
- 탐색 노드의 품질도 함께 올라갔다.

  **개선 전 — 불필요한 노드 포함, 수정 대상 노드는 얕게 탐색**

  **개선 후 — 수정 대상만 집중 탐색, 세부 구조까지 확보**

- 전체적으로 성공. 단순히 속도만 빨라진 게 아니라, 계획서에 담기는 Figma 분석 내용의 정확도와 밀도도 올라갔다.


### 시행착오 3 — Figma API가 왜 여러 번 호출되는지 의구심이 생겼다

워크플로우 구조를 바꾸고 나서도 한 가지 의문이 남았다. "Figma API가 왜 아직도 여러 번 호출되지?" 호출 횟수를 로그로 보다 보니, 노드 하나를 가져오는데 API가 여러 번 날아가는 패턴이 눈에 띄었다.

Figma REST API 공식 문서를 직접 찾아봤다. 확인해보니 skill에서 사용하고 있는 API로 **한 번의 API 호출로 하위 노드를 한꺼번에 가져올 수 있었다.**
근데 굳이, Agent가 노드를 하나씩 순차적으로 탐색하면서 매번 API를 호출하고 있었던 것이다. 스킬 프롬프트에 "분석 대상 노드를 한 번의 API 호출로 일괄 조회하라"는 지시를 추가했다.

**결과:** API 1번 호출로 필요한 노드를 모두 분석하는 데 성공했다.


## Learning (배움)

- **결정적 활용팁:** Agent 워크플로우를 빠르게 만들고 싶다면, 병렬화보다 "다음 단계의 입력 범위를 좁혀주는 구조"를 먼저 설계해야 한다.

- **배운 것:**
  1. **병렬 실행은 속도 개선의 일부일 뿐이며, 워크플로우 구조 설계가 더 중요하다.**
     단순히 두 작업을 동시에 돌리는 것보다, 어떤 순서로 컨텍스트를 쌓아서 다음 Agent에게 넘겨줄지를 설계하는 것이 성능에 훨씬 큰 영향을 미쳤다.

  2. **Agent에게 많은 정보를 주기보다 "분석 대상 범위를 줄이는 것"이 더 효과적이다.**
     컨텍스트가 많다고 결과가 좋아지지 않는다. 오히려 관계없는 정보가 많으면 Agent가 넓게 탐색하다가 진짜 중요한 부분을 얕게 보는 역효과가 생긴다.

  3. **공식 문서를 먼저 읽었다면, 바로 해결할 수 있었다.**
     Figma API가 여러 번 호출되는 문제는 공식 문서를 한 번만 읽어봤다면 바로 알 수 있는 내용이었다. 워크플로우 구조를 고민하기 전에, 사용하는 도구의 기본 스펙부터 확인하는 습관이 필요하다는 걸 느꼈다.
