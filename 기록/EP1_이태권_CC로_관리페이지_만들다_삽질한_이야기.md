# CC로 내부 관리페이지 만들다 삽질한 이야기

> **작성자:** 멤버A
> **날짜:** 2026-02-22
> **에피소드:** EP1 (1주차)

---

## Situation (상황)

- **문제 상황:** 소속 회사에서 사용하던 레거시 스프레드시트 기반 업무 데이터를 체계적으로 관리할 내부 관리 페이지가 필요했다. 사내에 개발자가 나 혼자뿐이라 빠르게 만들어야 했다.
- **사용 도구:** Claude Code (CLI)
- **배경:** 기존에 구글 시트로 관리하던 업무 데이터가 수백 건이 넘어가면서 검색, 필터링, 상태 관리가 불가능해졌다. 웹 프레임워크 기반 웹앱으로 내부 관리 페이지를 만들기로 했다.

## Task (과제)

- **목표:** Claude Code에게 내부 관리 페이지의 CRUD 기능(목록 조회, 상세 보기, 수정, 상태 변경)을 한 번에 지시해서 동작하는 페이지를 만든다.

## Action (행동)

### 시행착오
- 처음에 "관리 페이지를 만들어줘"라고 한 줄로 지시했더니, CC가 임의로 테이블 스키마를 만들고 UI도 자기 맘대로 구성했다. 기존 데이터 구조와 전혀 맞지 않았다.
- 수정을 요청할 때마다 이전 컨텍스트를 잃어버리고 다른 파일을 건드리거나, 이미 고친 부분을 다시 되돌리는 현상이 반복됐다.
- "이 부분만 고쳐줘"라고 해도 관련 없는 컴포넌트까지 수정해서 사이드이펙트가 발생했다.

### 해결 방법
- **스키마를 먼저 명시했다:** DB 테이블의 DDL을 그대로 프롬프트에 넣어주고 "이 스키마 기반으로 작업해"라고 지시했다.
- **파일 단위로 작업을 쪼갰다:** "이 파일만 수정해. 다른 파일은 건드리지 마"라고 매번 명시했다.
- **CLAUDE.md에 프로젝트 컨텍스트를 작성했다:** 프로젝트 구조, 사용 기술 스택, 컨벤션을 적어두니 CC가 맥락을 유지하는 비율이 올라갔다.

## Result (결과)

- 처음 한 줄 지시로 3시간 삽질한 것을, 스키마 명시 + 파일 단위 지시 + CLAUDE.md 조합으로 바꾸니 동일한 기능을 1시간 안에 완성했다.
- 사이드이펙트 발생 횟수가 체감상 절반 이하로 줄었다.
- 다만 복잡한 필터링 로직은 여전히 한 번에 잘 안 나와서 수동 수정이 필요했다.

## Learning (배움)

- **결정적 활용팁:** CC에게 "무엇을 만들어줘"보다 "이 제약 조건 안에서 이것만 해줘"가 훨씬 결과가 좋다. 제약 조건 = 스키마, 파일 범위, 컨벤션.
- **배운 것:** AI에게 자유도를 많이 줄수록 결과의 편차가 커진다. 명확한 경계(스키마, 파일 범위, 스타일 가이드)를 먼저 잡아주는 것이 "소통 양식"의 첫 번째 원칙인 것 같다. CLAUDE.md는 API 문서와 비슷한 역할을 한다.
