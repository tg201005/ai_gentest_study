## Situation (상황)

- **문제 상황:**
저희 팀은 Next.js로 FE·BE를 모두 구현하고 있는 상황입니다.
기존에는 Figma MCP를 활용해 "OO 기능 구현해줘"라고 요청하면 어느 정도 개발이 이루어졌지만, Figma MCP는 디자인 정보만 참조하여 개발을 수행하다 보니 기획 내용을 반영하지 못해 개발자가 수동으로 요청을 보완해야 하는 상황이 종종 발생했습니다.
Figma의 디자인과 Notion의 기획서를 함께 참조하여 개발 작업 TODO를 자동으로 생성하고, FE·BE 개발 전반을 자동화할 수 있을 것 같아 진행해보았습니다. 

- **사용도구**
Notion, Figma MCP, Claude SKILL

- **배경**
이미 Code Agent를 활용하여 Figma와 Notion의 내용을 수동으로 확인하며 개발하고 있던 상황이었습니다. 기존에 수행하던 작업 순서 그대로 Agent가 자동으로 처리할 수 있도록 설정해 두면 좋은 결과가 나올지 궁금했습니다.

## Task (과제)

- **목표:** Figma MCP, Notion (기획) 링크를 넘겨주면, 자동으로 수행할 TODO List를 만들고, 실행할 수 있는 환경 만들기

## Action (행동)

### 시행착오
**[첫 시도] 기본 환경 만들기**

1. skills 폴더에 Notion 기획 문서와 Figma 디자인을 불러와 구현 방향을 분석하는 analyze skill 생성
2. analyze가 산출한 문서를 기반으로 개발 TODO를 생성하는 plan skill 생성
3. plan이 산출한 문서를 기반으로 개발자가 요청한 TODO를 실행하는 build skill 생성

처음에는 "Figma MCP와 Notion을 활용해 구현 방향을 분석하고 계획하는 skill을 만들어줘"라는 대략적인 요구사항만 전달하여 Claude Code로 skill들을 생성했습니다.


**[시도 내 문제점]**

1. Figma 디자인 분석 결과를 기반으로 개발을 진행하면, Claude가 Figma 원본을 직접 참조하지 않고 분석 문서만을 참고하여 구현하여 디자인과 결과물이 일부 일치하지 않는 문제가 발생했습니다.
2. 일치하지 않은 디자인을 재요청할 경우, 매번 Figma 링크를 다시 분석하는 과정이 반복되어 재요청 시마다 처리 불필요한 조회 시간이 소요되는 문제가 있었습니다.
(e.g. 한 화면에 A, B, C, D 컴포넌트가 존재할 때, C 컴포넌트 디자인 확인을 재요청하면 A·B·C·D를 구분하는 작업부터 다시 수행한 후 C를 찾아 확인하므로, 불필요한 데이터 탐색 작업이 상당 부분을 차지했습니다.)


### 해결 방법
1. 배경색, 폰트 크기, 폰트 색상 등 Figma를 직접 열지 않아도 파악 가능한 스타일 요소는 문서 내에 명시하도록 프롬프트 추가
2. Figma 분석 Skill 내에서 컴포넌트 구조를 구분하여 분석하고, 각 컴포넌트의 Figma 노드 ID를 추출해 문서 내에 명시하도록 프롬프트 추가. 이를 통해 특정 컴포넌트의 디자인이 일치하지 않을 경우, 해당 컴포넌트의 노드만 추출하여 조회할 수 있도록 구성


## Result (결과)
간단한 기능 구현을 요청한 것이지만, 예상보다 계획도 잘 수립하고 디자인 요소도 잘 반영되었습니다. 다만 사소한 디자인 불일치로 인한 수정 재요청이 다소 발생했습니다.

개선 후에는 문서를 기반으로 해당 컴포넌트의 노드만 추출하여 확인하다 보니, 관련 작업 비용이 크게 줄었음을 체감했습니다.

## Learning (배움)

- **배운 것** : 외부 작업물을 분석하여 가공된 산출물을 생성하고, 이를 기반으로 구현하는 과정에서 Agent가 가공된 산출물이 아닌 원본을 직접 참조할 수 있도록 구성하는 것이 중요하다는 점을 배웠습니다.

- **결정적 활용 팁** : 아직 팀 내 개발 규칙이 충분히 마련되어 있지 않다 보니 자동화에 한계가 있다고 느꼈습니다. 추후 규칙과 테스트 로직이 잘 구성된다면 간단한 작업은 더욱 자동화할 수 있을 것으로 생각됩니다.
